Reasoning for Selection of User Story and Estimation

For Sprint 0 our group has chosen to implement User Story #17 (Create a new event) for three explicit reasons. Firstly, we want to implement administration applications before member applications. The reason being, we have direct contact with the users who will have administration rights (our product champion). This way, we will be able to quickly, and easily, learn, and receive, direct feedback of any preferences they might have.

Secondly, nearly all our user stories require interaction with a database to insert or retrieve information. Completing a user story that will successfully interact with the database will provide a good platform for other stories to be developed in parallel.

Lastly, this user story has been estimated to be much simpler than many of the other stories for this project (such as ones that include email automation, paypal payments, and database table modifications), estimated to be a two (look to Susanâ€™s notes for scale). The size of this story fits the time frame given considering all members needed to set-up, install, and learn ruby on rails.
